Webhook Delivery

Send your iPhone's notification and call events directly to any service that accepts a JSON POST — no Home Assistant required. Here's how it works, what the payloads look like, and where to send them.

What & why

ANCService already delivers notification and call events to Home Assistant as homeassistant.event actions. Webhooks are a second, parallel delivery path: the device sends the same event data as a JSON POST request straight to a URL you choose — to a self-hosted automation tool, a notification relay like ntfy.sh, or a custom receiver. Use webhooks if you don't run Home Assistant, or want to feed notification data into something else entirely. Both delivery paths can run side by side.

How to configure

Webhook delivery is off by default and entirely optional. You'll find the controls grouped under Webhooks — on the device's local web page (browse to its IP address) and under Configuration in Home Assistant:

  • Webhooks Enabled — a master switch. Nothing is sent while it's off, regardless of which URLs are set.
  • Notification Webhook URL — where to POST regular notifications.
  • Incoming Call Webhook URL — where to POST incoming-call events.
  • Call Ended Webhook URL — where to POST call-ended events.

Each URL is independent and optional — leave one blank and that event type simply won't be sent, even with the master switch on. Set just the one(s) you need.

Payload reference

Each webhook sends a JSON body with Content-Type: application/json, mirroring the fields sent to Home Assistant for the same event:

WebhookFields
Notification Webhook URL uid, category, app_id, title, subtitle, message, date, device_name, iphone_name
Incoming Call Webhook URL uid, caller, app_id, date, device_name, iphone_name
Call Ended Webhook URL uid, device_name, iphone_name

Example — an incoming-call POST body:

{
  "uid": "1234",
  "caller": "Jane Appleseed",
  "app_id": "com.apple.mobilephone",
  "date": "2026-06-07T14:32:00Z",
  "device_name": "ancservice-a1b2c3",
  "iphone_name": "Jane's iPhone"
}
Example — a regular notification POST body 
{
  "uid": "5678",
  "category": "other",
  "app_id": "com.apple.MobileSMS",
  "title": "Jane Appleseed",
  "subtitle": "",
  "message": "Running 5 minutes late!",
  "date": "2026-06-07T14:35:00Z",
  "device_name": "ancservice-a1b2c3",
  "iphone_name": "Jane's iPhone"
}
Example — a call-ended POST body 
{
  "uid": "1234",
  "device_name": "ancservice-a1b2c3",
  "iphone_name": "Jane's iPhone"
}

Where to send it

Any URL that accepts a JSON POST works. A few starting points:

  • ntfy.sh — free push notifications to your phone or desktop; POST straight to a topic URL.
  • webhook.site or RequestBin — see exactly what arrives, great for testing before wiring up the real thing.
  • Your own automation tool (Node-RED, n8n, Home Assistant's webhook trigger, etc.) — anything with an HTTP-accessible endpoint.

Need authentication? There's no separate header field in v1 — embed a token directly in the URL as a query parameter (e.g. https://example.com/hook?token=secret), which ntfy.sh, Discord, and most generic receivers accept.

Troubleshooting

Nothing arriving? Work through this checklist:

  1. Is Webhooks Enabled switched on?
  2. Is the URL for that specific event type set (and spelled correctly, including https:// or http://)?
  3. Check the device logs (ESPHome dashboard → device → Logs, or the local web page's log stream) for http_request entries — failed deliveries (timeouts, DNS errors, non-2xx responses) are logged there.
  4. Try pointing the URL at webhook.site first to confirm the device can reach the internet and is sending what you expect.
