MQTT Delivery

Publish your iPhone's notification and call events straight to MQTT topics on your own broker — no Home Assistant or webhook receiver required. This is an advanced, build-your-own-firmware path: here's what it involves, what the payloads look like, and how to set it up.

What & why — and the catch

ANCService already delivers notification and call events to Home Assistant as homeassistant.event actions, and can send the same data as JSON webhooks. MQTT delivery is a third path: the device publishes the same event data as JSON messages directly to topics on an MQTT broker — useful if you already run a broker (standalone, or as part of Home Assistant) and want native topic-based integration, MQTT discovery, or to feed the data into existing MQTT-based automations.

The catch: unlike webhooks, this isn't a toggle you can flip in the curated app. ESPHome's MQTT client needs to know your broker's address at build time — it can't be configured later from the web UI or Home Assistant. So MQTT support ships as a separate firmware variant, firmware/ancservice-mqtt.yaml, that you build and flash yourself with your broker details filled in. If that sounds like more than you want to take on, webhook delivery covers the "I don't run Home Assistant" use case with zero building required — see the webhook delivery guide.

How to build it

Building the MQTT variant means compiling and flashing your own firmware:

  1. Clone the ANCService repo.
  2. Open firmware/ancservice-mqtt.yaml and edit the mqtt: block — fill in your broker's host/IP and username where marked EDIT ME.
  3. Add your broker password to your own secrets.yaml (create one in the firmware/ directory if you don't have one — it's gitignored, so it stays local and is never committed): 
    mqtt_password: "yourpassword"
  4. Compile and flash with ESPHome: 
    esphome run firmware/ancservice-mqtt.yaml

New to ESPHome's build/flash process? Its getting started guide covers the basics. For the full set of broker options this minimal setup doesn't expose — TLS, custom ports, Home Assistant MQTT discovery, and more — see ESPHome's mqtt: component reference.

Runtime control

Once flashed, MQTT delivery is controlled by a single switch — grouped under MQTT on the device's local web page (browse to its IP address) and under Configuration in Home Assistant:

  • MQTT Enabled — a master switch. Off by default; nothing publishes until you turn it on. There's no per-event-type control — it's all three event types together, or none.

Payload & topic reference

Each event publishes a JSON message (QoS 0, not retained) to a fixed topic derived from your device's name, mirroring the same fields sent to Home Assistant and webhooks for that event:

TopicFields
ancservice/<device>/notification uid, category, app_id, title, subtitle, message, date, device_name, iphone_name
ancservice/<device>/incoming_call uid, caller, app_id, date, device_name, iphone_name
ancservice/<device>/call_ended uid, device_name, iphone_name

<device> is your device's full name, including its MAC suffix (e.g. ancservice-a1b2c3) — the same identity used as the device_name field inside the payloads themselves, so topics and payload contents stay consistent.

Example — an incoming-call message on ancservice/ancservice-a1b2c3/incoming_call:

{
  "uid": "1234",
  "caller": "Jane Appleseed",
  "app_id": "com.apple.mobilephone",
  "date": "2026-06-07T14:32:00Z",
  "device_name": "ancservice-a1b2c3",
  "iphone_name": "Jane's iPhone"
}
Example — a regular notification message 
{
  "uid": "5678",
  "category": "other",
  "app_id": "com.apple.MobileSMS",
  "title": "Jane Appleseed",
  "subtitle": "",
  "message": "Running 5 minutes late!",
  "date": "2026-06-07T14:35:00Z",
  "device_name": "ancservice-a1b2c3",
  "iphone_name": "Jane's iPhone"
}
Example — a call-ended message 
{
  "uid": "1234",
  "device_name": "ancservice-a1b2c3",
  "iphone_name": "Jane's iPhone"
}

Troubleshooting

Nothing arriving? Work through this checklist:

  1. Is MQTT Enabled switched on?
  2. Check the device logs (ESPHome dashboard → device → Logs, or the local web page's log stream) for mqtt entries — connection failures (wrong broker address, bad credentials, network issues) are logged there as the client attempts to connect.
  3. Confirm your broker details in firmware/ancservice-mqtt.yaml and secrets.yaml are correct, then rebuild and reflash — remember, broker settings are baked in at build time and can't be changed from the web UI afterward.
  4. Subscribe directly with a tool like mosquitto_sub to confirm messages are arriving at the broker: 
    mosquitto_sub -h yourbroker -t 'ancservice/#' -v
